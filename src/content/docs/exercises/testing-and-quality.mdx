These exercises cover the production-quality concerns from Phases 5–7. They assume you're already using Copilot and Claude regularly and want to make that usage more rigorous.

Each exercise takes 20–40 minutes. They are independent — do them in any order based on what's most relevant to your current work.

---

## Before you start

You need:

- Claude (web or Code) and Copilot active
- A codebase with existing tests (at least some) so you have patterns to match
- Read: [AI-Assisted Testing](/testing/ai-assisted-testing/), [Reviewing AI-Generated Code for Security](/security/reviewing-ai-generated-code/), [AI for Pipeline Configuration](/cicd/ai-for-pipeline-config/)

## Exercise 1: Test scaffolding with real assertions

**What you'll practice:** Using AI for test structure, not for test correctness — and writing the assertions that actually matter yourself.

Find a method or function in your codebase that has no tests or thin coverage. Pick something you understand well — you need to be able to evaluate whether the generated tests are right.

**Step 1 — Generate structure:**

Ask Claude or Copilot to generate a test file for this method. Include the method code and any relevant dependencies. Specify your testing library and ask it to match the patterns in an existing test file.

Read the generated output before running it.

**Step 2 — Audit what was generated:**

For each test, ask:

- Does this test assert the correct outcome, or just the current behaviour?
- Would this test fail if the method had a bug in the most likely place?
- Is the mock setup realistic — does it match what the real dependency would actually return?

Mark any test you're not confident about.

**Step 3 — Ask for missing cases:**

Tell Claude: _"Here are the business rules this method must enforce: [list them]. What test cases are missing from the generated file?"_

Evaluate the suggestions. Add the ones that genuinely matter.

**Step 4 — Write the assertions yourself:**

For the tests covering business rules, replace the AI-generated assertion with one you wrote. Confirm it fails when you introduce a deliberate bug (comment out a guard clause, change a comparison operator, return a wrong value). If the test passes with the bug, the assertion is too weak.

<Callout type="warning">
  A test suite that always passes — including when the code is wrong — is worse than no tests. It creates false
  confidence. The only assertion worth trusting is one you've seen fail.
</Callout>

## Exercise 2: Security review of AI-generated code

**What you'll practice:** Using the OWASP-pattern checklist to review something Copilot or Claude actually wrote for you.

**Reference:** [Reviewing AI-Generated Code for Security](/security/reviewing-ai-generated-code/)

Pick a piece of code that was generated by AI in the past month — an endpoint, a service method, a data access function. If you don't have recent AI-generated code, ask Copilot to generate a basic CRUD endpoint for any entity in your codebase.

Run through the checklist from the security guide manually. For each item, mark whether it applies to this code and whether the AI-generated version handles it:

- **Input validation:** Is every externally-supplied value validated before use?
- **SQL / query injection:** If there's a database call, are parameters bound — not concatenated?
- **Authentication and authorisation:** If this is an endpoint, does it check who's calling and what they're allowed to do?
- **Sensitive data:** Are secrets, tokens, or personally-identifying data logged or returned in error messages?
- **Error handling:** Does the error response leak internal details — stack traces, file paths, SQL queries?

After the manual review, paste the same code into Claude and ask: _"Review this code for common security issues. Be specific about which lines are problematic and why."_

Compare the two reviews. What did Claude catch that you missed? What did you catch that Claude missed or understated? What did Claude flag as a concern that isn't actually a risk in your context?

## Exercise 3: Generate and evaluate a CI/CD workflow

**What you'll practice:** Using AI to draft pipeline configuration, then evaluating it against what your team actually needs.

**Reference:** [AI for Pipeline Configuration](/cicd/ai-for-pipeline-config/)

Describe your project's build and test requirements to Claude in plain language. Include:

- The language and framework
- What "build" means (compile, lint, bundle, etc.)
- What "test" means (unit tests, integration tests, which ones are slow)
- Any environment variables or secrets the pipeline needs
- Which branch triggers which behaviour (e.g., PR vs. merge to main)

Ask Claude to generate a GitHub Actions workflow file for this.

Evaluate the output against these questions:

1. **Does it build correctly?** Check the specific commands it uses against your actual build process. AI often generates commands for the default project structure, not yours.

2. **Does it handle secrets correctly?** Are secrets referenced as `${{ secrets.NAME }}` rather than hardcoded? Is there anything sensitive in the step names or log output that would appear in public CI logs?

3. **Is the trigger correct?** Does it run on the events you want? Does it exclude anything it shouldn't (e.g., only running on `main` when you want it on all PRs)?

4. **Are there unnecessary steps?** AI often generates maximal workflows — caching, matrix builds, artefact uploads — even when you didn't ask. Identify what you'd remove.

Make the changes needed to make it accurate. The goal isn't to generate a perfect workflow — it's to generate a usable starting point in five minutes that would have taken an hour from scratch.

## Exercise 4: Catch a hallucinated dependency

**What you'll practice:** Identifying when AI has invented a library, method, or API that doesn't exist — before it makes it into your codebase.

Ask Copilot or Claude to generate something that involves an external library — a new test helper using your testing framework, a service that calls a third-party API, a utility using a package you know exists but don't use constantly.

Specifically look for:

- Method names you don't recognise — look them up in the official documentation before using them
- Constructor signatures or parameter names that don't match the current version
- Import or namespace paths that seem slightly off
- Package names that sound plausible but that you haven't verified exist

For any method or class you can't immediately confirm from memory, look it up. Don't run the code until you've confirmed each external name is real.

If you find a hallucination: note what it was, how plausible it looked, and what confirmed it was wrong (error on run, not in docs, wrong signature). This builds the habit of treating AI method suggestions as drafts, not facts.

---

## What good looks like

After these exercises, you should be able to:

- Generate test scaffolding quickly and know which assertions to trust vs. rewrite
- Run a security review pass on AI-generated code before it goes to PR
- Draft and evaluate a CI/CD workflow without starting from scratch
- Catch hallucinated methods and imports before they cause build failures
