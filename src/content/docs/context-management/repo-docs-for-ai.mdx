AI tools are only as useful as the context they receive. For a small, solo project, this isn't much of a problem вЂ” the whole codebase fits in a context window. For a team working on a multi-service backend, the problem compounds: every developer is re-pasting the same architectural conventions into every AI session, context gets stale, and the AI makes suggestions that are technically correct but architecturally wrong.

The solution is treating AI-readable documentation as a first-class concern вЂ” not an afterthought.

## The three types of AI-useful documentation

AI tools consume documentation differently from humans. Humans read sequentially and infer context from surrounding text. AI tools work from whatever is in their context window right now.

This means the most useful documentation for AI tools is:

1. **Orientation docs** вЂ” "What does this service do, what are its boundaries, what does it depend on?" A human reads this once. An AI needs it at the start of every relevant session.

2. **Convention docs** вЂ” "How is code structured here? What patterns are in use? What should never happen?" These are the rules an AI follows when generating or reviewing code.

3. **Decision docs (ADRs)** вЂ” "Why did we make this choice?" These prevent AI tools from helpfully replacing a pattern they don't understand with something generic.

## AGENTS.md вЂ” per-service context files

The canonical pattern for per-service AI context is an `AGENTS.md` file in the service directory. Claude Code reads this file automatically when working in that directory. Copilot Agent Mode picks it up when exploring. It's also referenced from the top-level `.github/copilot-instructions.md` (see the [copilot-instructions guide](/context-management/copilot-instructions-file/)).

<Callout type="note">
  `AGENTS.md` and `CLAUDE.md` are related but distinct. AGENTS.md is a cross-tool open standard read by Claude Code, Copilot, Cursor, and others вЂ” the right choice for multi-tool teams. CLAUDE.md is Claude Code's native format with additional features (subdirectory hierarchy, `@import` support, auto-memory). Claude Code reads both; CLAUDE.md takes priority when both exist. See [CLAUDE.md vs AGENTS.md](/context-management/agent-files-explained/) for the full breakdown and decision guide.
</Callout>

### What to put in a service-level AGENTS.md

Keep it short, factual, and opinionated. The goal is to give an AI the same orientation a competent new team member would need on their first day working on this service.

```markdown
# OrderService вЂ” Agent Context

## Purpose
Handles order lifecycle: creation, status transitions, cancellation, and fulfilment
notification. Does NOT handle payments (see PaymentService) or inventory (see
InventoryService).

## Stack
- ASP.NET Core 8 Web API
- Entity Framework Core 8 with PostgreSQL
- Azure Service Bus for event publishing
- MediatR for CQRS (commands and queries are strictly separated)

## Key patterns
- All data access through `IOrderRepository` вЂ” no direct DbContext access outside
  the repository layer.
- Handlers return `Result<T>` вЂ” never throw for domain validation failures;
  throw only for infrastructure failures.
- Events published via `IEventPublisher` after successful command handling.
- Tests: xUnit, Moq, TestContainers for integration tests.

## Naming conventions
- Commands: `{Verb}OrderCommand` (e.g. `CreateOrderCommand`, `CancelOrderCommand`)
- Queries: `Get{Thing}Query` (e.g. `GetOrderByIdQuery`, `GetOrderHistoryQuery`)
- Handlers: `{CommandOrQuery}Handler`

## What NOT to do
- Do not add direct DbContext injection to controllers or handlers.
- Do not throw `ArgumentException` for validation errors вЂ” use `Result.Failure`.
- Do not publish events from the repository layer.

## Related context
- Architecture overview: /docs/architecture/services/order-service.md
- API contract: /docs/api/order-service-openapi.yaml
- ADR-003: Why we chose CQRS for this service: /docs/decisions/ADR-003-cqrs.md
```

The critical sections: **Purpose** (what it does and doesn't do), **Stack** (exact libraries), **Key patterns** (the rules), **What NOT to do** (the constraints), **Related context** (links to more detail).

### What to leave out

- Business rules that change frequently
- Anything that's obvious from the code itself
- Prose explanations of standard .NET patterns (the AI already knows what EF Core is вЂ” you just need to say you're using it)
- Details that belong in a different service's AGENTS.md

## Architecture docs as orientation context

Alongside per-service AGENTS.md files, a `/docs/architecture/` directory with brief service overviews gives AI tools a map of the whole system.

Each service overview should answer:
- What is this service responsible for?
- What does it depend on?
- What depends on it?
- How does it communicate (REST, event bus, gRPC)?

Example (`/docs/architecture/services/order-service.md`):

```markdown
# OrderService

**Owns:** Order lifecycle (creation в†’ fulfilment в†’ cancellation)
**Does not own:** Payment processing, inventory management

**Upstream dependencies:**
- UserService (validates customer identity via REST)
- InventoryService (checks stock via REST before order creation)

**Downstream consumers:**
- FulfilmentService (subscribes to `OrderConfirmed` event via Service Bus)
- NotificationService (subscribes to `OrderStatusChanged` event)

**API surface:** REST (internal, behind API Gateway) + Azure Service Bus events

**Database:** PostgreSQL, `orders` schema, owned migrations
```

This is information no AI can infer from the code. With it, you can ask "I'm adding a feature to OrderService вЂ” what other services might I need to coordinate with?" and get a useful answer.

## ADRs вЂ” why the pattern exists

Architecture Decision Records prevent AI tools from replacing established patterns with something "better." Without an ADR explaining why you chose the repository pattern, an AI seeing your service might suggest accessing the database directly in a handler вЂ” it's less code and technically works.

An ADR explains the context, decision, and consequences:

```markdown
# ADR-003: CQRS for OrderService

## Status: Accepted

## Context
OrderService started as a simple CRUD service. As the domain grew, we needed:
- Different read models for the UI vs internal integrations
- The ability to audit every state transition
- Clearer separation of side effects (event publishing) from reads

## Decision
Use MediatR to implement CQRS. Commands mutate state and publish events.
Queries return read-optimised DTOs. Handlers are the unit of work.

## Consequences
- More files per feature (command + handler + query + handler)
- Clear place for every behaviour
- Direct DbContext access becomes obviously wrong вЂ” it bypasses the CQRS boundary
- New developers need to understand CQRS before contributing
```

When an AI references or reads this ADR, it understands *why* the pattern exists and won't suggest violating it without understanding the trade-off.

<Callout type="tip">
  Link ADRs from the relevant AGENTS.md file. An AI working on a service is more likely to encounter the ADR when it's explicitly referenced than when it has to discover it by searching.
</Callout>

## Keeping it current

Documentation that's wrong is worse than no documentation вЂ” it actively misleads the AI.

Two rules that help:

**Update in the same PR.** If a PR changes a pattern (e.g., switches from throwing exceptions to returning `Result<T>`), the `AGENTS.md` update is part of that PR. Make it a PR template checklist item.

**ADRs are append-only.** Don't edit an existing ADR when a decision changes вЂ” add a new one that supersedes it, and mark the old one as "Superseded by ADR-007." This preserves the history an AI might reference and makes the evolution traceable.
