Every time you paste code into Copilot Chat or Claude, you're sending it to an external service. In most contexts this is fine — you're sharing generic code patterns, not sensitive data. The risk comes when config files, environment variables, connection strings, or other sensitive values end up in the context without you noticing.

This guide covers what to watch for and the habits that prevent accidental exposure.

---

## What counts as sensitive in a prompt

**Credentials and secrets:**

- Connection strings (database URLs with passwords)
- API keys and tokens
- OAuth client secrets
- Private keys and certificates
- Passwords in any form

**Internal infrastructure:**

- Internal hostnames and IP addresses
- Service endpoint URLs for internal systems
- VPN or network configuration details
- Infrastructure topology that represents an attack surface

**Regulated data:**

- Names, email addresses, phone numbers (PII)
- Payment card data
- Health records or any regulated category
- Any data that would be a problem if it appeared in a log file

**Organisational context:**

- Internal system names and codenames
- Customer names or business relationships
- Unreleased product details

<Callout type="warning">
  The most common accidental exposure is a config file or `.env` file included alongside the code you meant to share.
  Before pasting into any AI tool, check what's in the files you're including.
</Callout>

## Sanitising code before sharing

When you need AI help with code that touches sensitive configuration, replace the real values with placeholders before pasting:

**Before (don't paste this):**

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=prod-db.internal.company.com;Database=OrdersDB;User Id=svc_orders;Password=Xk9$mPq2;"
  },
  "ApiKeys": {
    "PaymentGateway": "live_sk_abc123xyz789"
  }
}
```

**After (safe to paste):**

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=<DB_HOST>;Database=<DB_NAME>;User Id=<DB_USER>;Password=<DB_PASSWORD>;"
  },
  "ApiKeys": {
    "PaymentGateway": "<PAYMENT_GATEWAY_API_KEY>"
  }
}
```

The AI gets enough context to help you with the structure and patterns. The sensitive values aren't transmitted.

For code files that reference configuration by key name (not by value), there's usually no issue — sharing `_configuration["ConnectionStrings:DefaultConnection"]` is fine.

## The `.env` file risk

`.env` files contain real secrets for local development. They should be in `.gitignore` and should never appear in a prompt.

The risk isn't intentional — it's that you ask AI to "look at my config" and share the wrong file. Check:

```
# .gitignore — these should all be present
.env
.env.local
.env.*.local
local.settings.json    # Azure Functions local config
appsettings.Local.json # ASP.NET Core local override
```

If any of these are tracked in git, remove them immediately:

```bash
git rm --cached .env
echo ".env" >> .gitignore
git commit -m "Remove .env from tracking"
```

## Claude and Copilot data handling

**GitHub Copilot:**

- For organisations with a GitHub Enterprise or Copilot Business subscription, prompts are not used to train the model by default
- Check your organisation's Copilot policy settings to confirm
- Individual (personal) Copilot subscriptions have different defaults — check the privacy settings in your GitHub account

**Claude (Anthropic):**

- Claude.ai (web interface): conversations may be used to improve models depending on your plan and settings; check the privacy settings for your account
- Claude API: by default, API inputs are not used to train models
- If your team accesses Claude via API in pipelines, API traffic is not used for training

For regulated industries or when handling particularly sensitive work, confirm your organisation's policy on which tools are approved before using any AI tool with production-adjacent code.

## Configuration hygiene that prevents accidents

The best protection is a project structure where secrets are never in source files in the first place.

**For .NET: use environment variables or Azure Key Vault**

```csharp
// appsettings.json — committed, no secrets
{
  "Database": {
    "ConnectionStringName": "DefaultConnection"
  }
}

// appsettings.Development.json — in .gitignore
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost;..."
  }
}
```

In CI/CD, inject secrets via GitHub Actions secrets or Azure Key Vault references — never in the workflow YAML directly.

**For React: use environment variables with a naming convention**

```
# .env.example — committed, shows structure with no values
VITE_API_BASE_URL=
VITE_ANALYTICS_KEY=

# .env.local — in .gitignore, actual values for local dev
VITE_API_BASE_URL=http://localhost:5000
VITE_ANALYTICS_KEY=dev_key_placeholder
```

Commit `.env.example` (the structure), never `.env.local` (the values). When a new developer joins, they copy `.env.example` to `.env.local` and fill in the values.

## Pre-commit hooks as a safety net

A pre-commit hook that scans for common secret patterns adds a layer of protection:

```bash
# Install: npm install --save-dev @secretlint/secretlint secretlint
# .secretlintrc.json
{
  "rules": [{ "id": "@secretlint/secretlint-rule-preset-recommend" }]
}
```

Add to your pre-commit hook:

```bash
npx secretlint "**/*"
```

This catches common patterns — AWS keys, connection strings, private key headers — before they're committed. It doesn't catch everything, but it catches the obvious mistakes.
