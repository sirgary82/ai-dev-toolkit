AI tools are genuinely good at writing test scaffolding. They can generate `describe`/`it` blocks, boilerplate setup and teardown, and repetitive assertion patterns faster than any developer. The problem is mistaking scaffolding for a test strategy. A file full of AI-generated tests that all pass isn't the same as a file full of tests that will catch real failures.

This guide covers how to use Copilot and Claude for testing effectively — and where to draw the line.

---

## What AI does well in testing

**Generating test structure from existing code.** Give Copilot or Claude an existing function or class, and they'll produce a test file with plausible structure: setup, happy path, a few sad paths. For a `UserService` with five methods, you'd spend more time creating the file and writing the boilerplate than writing the first tests.

**Naming cases you haven't thought of.** This is underrated. Ask Claude: *"Here is this function and its expected behaviour. What test cases am I missing?"* It will generate a list that includes boundary conditions, null inputs, empty collections, and concurrent-access scenarios — cases that are easy to skip when you're writing tests after writing code.

**Repetitive parameterised tests.** Writing a test that checks ten different input/output combinations? AI can generate the data table. You verify the cases are correct; AI writes the repetition.

**Test for code you didn't write.** When you inherit a class with no tests, asking Claude to explore it and generate a test suite gives you a starting point. The tests won't be right until you review them — but starting from something is faster than starting from nothing.

---

## The limit: AI doesn't know your business rules

AI generates tests from what it can see: the function signature, parameter names, return types, and any obvious logic. It does not know:

- What the function's output *means* in your domain
- Which edge cases actually matter for your users
- What a "valid" state looks like for your business objects
- Which paths carry the highest production risk

A function named `calculateDiscount` will get AI-generated tests for null inputs, negative values, and zero. What it won't get are tests for the specific rules your business applies — a rule that certain customer tiers always get a minimum 10%, or that discounts can't stack above a certain threshold. Those rules live in your head and your domain documentation, not in the code.

The consequence: AI-generated tests tend to assert the code's current behaviour rather than the intended behaviour. If the code has a bug, the AI-generated test will often test the buggy behaviour and pass.

<Callout type="warning">
  Never accept AI-generated tests without reading every assertion. The question isn't "do these tests pass?" — it's "do these tests fail when the code does the wrong thing?"
</Callout>

## The pattern: AI writes scaffolding, you write the assertions that matter

**Step 1 — Generate the structure:**

```
Here is OrderService.cs. Generate a test file for the PlaceOrder method.
Include setup, a happy path test, and stubs for the main failure cases.
Use xUnit and Moq, matching the patterns in the existing test files.
```

AI produces a file with the right shape: class, constructor, test methods, mock setup.

**Step 2 — Ask for the cases you might be missing:**

```
Given that PlaceOrder must enforce these business rules:
- Order total must be positive
- Customer must not have an outstanding balance
- Inventory must be reserved before the order is confirmed

What test cases am I missing from this test file?
```

Read the list. Some will be obvious; some will be genuinely useful.

**Step 3 — Write the assertions yourself.**

The AI-generated test stubs have the right *structure* — the arrange/act/assert skeleton. Fill in the assertions for the cases that matter. For business-rule tests, you write those from scratch.

**Step 4 — Run and fix.**

AI-generated mock setups are often slightly wrong — a method name off by one character, a return type mismatch. Fix these as you go. The value is that you're fixing small errors rather than writing from zero.

## Prompts that work

**Edge case generation:**
```
Here is this function and its signature. List every edge case and boundary
condition you would test, including inputs that should be rejected and
any failure modes in the dependencies.
```

**Test review:**
```
Read these tests and the code they test. Which tests would pass even if
the code had a bug? Which assertions are too weak to be useful?
```

**Coverage gap analysis:**
```
Here is the code and the existing test file. What branches or paths in
the code have no corresponding test?
```

**Existing test explanation:**
```
Explain what this test is actually asserting and whether the assertion
is strong enough to catch a regression.
```

## For .NET (xUnit + Moq)

When generating tests, include the existing test file as context so AI matches your patterns:

```
@ExistingServiceTests.cs
Generate tests for OrderService.PlaceOrder following the same patterns —
xUnit Facts and Theories, Moq for dependencies, AAA structure.
```

AI will match your mock setup style, your assertion library (FluentAssertions vs raw Assert), and your naming convention.

## For React (React Testing Library)

Provide the component and any related stories or prop types:

```
@ProductCard.tsx @ProductCard.stories.tsx
Generate tests for ProductCard using React Testing Library.
Test the rendered output and user interactions, not implementation details.
```

The instruction "not implementation details" matters — AI will default to testing internal state if you don't specify.
